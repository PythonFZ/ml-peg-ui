# Phase 01.1: Replace ORJSONResponse with Pydantic Response Models - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace all `ORJSONResponse` usage with typed Pydantic response models across the FastAPI API. Remove the `orjson` dependency entirely (both response serialization and storage parsing). FastAPI's built-in Pydantic serialization handles JSON output. This is a focused refactor — no new endpoints, no new features.

</domain>

<decisions>
## Implementation Decisions

### Model granularity
- Per-endpoint Pydantic response models: HealthResponse, CategoriesResponse, BenchmarkTableResponse, ModelsResponse
- Shared `Meta` base model (count: int, columns: list[str] | None = None) composed into each response model
- Health endpoint stays simple (`{"status": "ok"}`) — no envelope wrapping
- All models live in a single `api/models.py` file

### Table row typing
- `BenchmarkRow` model with typed base fields: `id: str`, `MLIP: str`, `Score: float | None = None`
- Dynamic metric columns handled via `model_config = ConfigDict(extra="allow")`
- Extra fields passed through as-is (Any type, no validation/coercion) — data is pre-computed and trusted
- `meta.columns` lists ALL columns including id/MLIP/Score — frontend uses it directly for DataGrid column config

### Serialization strategy
- Remove `ORJSONResponse` entirely — it is deprecated in modern FastAPI
- FastAPI auto-serializes Pydantic models to JSON bytes when return type annotations are set
- Remove `orjson` dependency completely — replace `orjson.loads` in storage.py with `json.loads`
- Endpoints return Pydantic model instances with return type annotations (e.g., `-> CategoriesResponse`)
- No `response_class` parameter needed on any endpoint

### OpenAPI/docs output
- Types-only approach — Pydantic models auto-improve OpenAPI docs via type annotations
- No manual examples, Field(example=...), or json_schema_extra
- No endpoint tags or custom descriptions beyond what's already in docstrings
- 501 figure stub stays as HTTPException — will be replaced when implemented

### Test updates
- Update existing API tests to validate responses through Pydantic models (model_validate on response JSON)
- Catches schema drift between endpoint returns and model definitions

### Claude's Discretion
- Exact model field ordering and naming conventions
- Whether to use `model_validate` or `model_construct` in tests
- Import organization in api/models.py
- Any helper methods on the models (if useful)

</decisions>

<specifics>
## Specific Ideas

- ORJSONResponse is officially deprecated in FastAPI — FastAPI now serializes data directly to JSON bytes via Pydantic when a return type or response model is set, which is faster
- The `envelope()` helper function in index.py should be replaced by constructing Pydantic response model instances directly

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `envelope()` function in api/index.py: wraps data in `{"data": ..., "meta": {"count": N, ...}}` — will be replaced by Pydantic model constructors
- `StorageBackend` protocol in api/storage.py: uses `orjson.loads` — needs migration to `json.loads`

### Established Patterns
- All 5 endpoints use `response_class=ORJSONResponse` and return `dict` — all need updating
- `app = FastAPI(default_response_class=ORJSONResponse)` — remove this parameter
- Envelope pattern `{"data": ..., "meta": {...}}` is consistent across data endpoints — Pydantic models should preserve this exact shape

### Integration Points
- api/index.py: 5 endpoint functions + `envelope()` helper + FastAPI app constructor
- api/storage.py: `orjson.loads` in FilesystemBackend.get_json and MinioBackend.get_json
- tests/: existing API tests assert on raw JSON dicts — update to validate through models
- pyproject.toml: remove orjson from dependencies

</code_context>
